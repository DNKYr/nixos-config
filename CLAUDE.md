# nixos-config

> DNKYr's NixOS flake configuration for the `aether` laptop, managing system and home-manager declaratively.

---

## Quick Reference

| Task | Command |
|------|---------|
| Rebuild & switch | `sudo nixos-rebuild switch` (from `/home/dnkyr/nixos-config`) |
| Test without switching | `sudo nixos-rebuild test` |
| Build without activating | `sudo nixos-rebuild build` |
| Update flake inputs | `nix flake update` |
| Garbage collect | `sudo nix-collect-garbage -d` (automated weekly via `optimizeStore.nix`) |

**Non-negotiables:**
- Never edit `laptop/hardware-configuration.nix` — it is auto-generated by nixos hardware scan
- Always declare `inputs` in module function args when the module needs a flake input (e.g., `nix-gaming`, `zen-browser`, `noctalia`) — forgetting this causes an eval error
- Never bump `home.stateVersion` or `system.stateVersion` without reading the NixOS release notes — these are one-way migrations that can break stateful data
- Raw configs (`configs/niri/`, `configs/nvim/`) are symlinked by home-manager; edit the source in `configs/`, not the deployed path in `~/.config/`

## Project Overview

A Nix flakes-based NixOS configuration for a single machine (`aether`). System-level config (boot, networking, hardware) lives under `laptop/`. User environment (packages, dotfiles, shell) is managed by home-manager under `home/`. External programs with no nixpkgs packages are pulled in via flake inputs (noctalia shell, zen-browser, nix-gaming).

**Stack:** Nix flakes, NixOS unstable, home-manager, Niri (Wayland compositor), Zsh + Nushell, Zellij, Neovim (AstroNvim)
**Entry point:** `flake.nix` → `nixosConfigurations.aether`

## Architecture

```
flake.nix (nixosConfigurations.aether)
    │
    ├── optimizeStore.nix         (GC policy, store dedup)
    ├── modules/                  (NixOS system-level modules)
    │     ├── niri.nix            (Niri Wayland compositor)
    │     └── fonts/font.nix      (system font packages + fontconfig defaults)
    ├── laptop/
    │     ├── configuration.nix   (boot, networking, users, hardware services)
    │     └── hardware-configuration.nix  ← AUTO-GENERATED, never edit
    └── home-manager (home/)
          ├── default.nix         (packages, git, xdg symlinks, direnv)
          └── modules/
                ├── neovim.nix    (neovim + LSP packages)
                ├── shell/        (zsh, nushell, starship, alacritty, kitty, zellij)
                ├── noctalia.nix  (desktop shell via noctalia flake input)
                ├── zen-browser.nix  (browser + extensions via zen-browser flake)
                ├── claude.nix    (Claude Code)
                ├── cpp-dev.nix   (gcc, gdb, cmake, gnumake)
                └── gaming.nix    (osu-lazer via nix-gaming flake)

configs/  (raw dotfiles, symlinked via xdg.configFile in home/default.nix)
    ├── niri/config.kdl
    └── nvim/  (AstroNvim lua config)
```

**Key invariants:**
- `modules/` contains only NixOS system-level options; user-space belongs in `home/modules/`
- `configs/` contains only raw non-Nix config files; Nix-managed program settings go inline in the `.nix` module
- `flake.nix` passes `inputs` via `specialArgs` (system) and `extraSpecialArgs` (home-manager) — all modules receive it

## Coding Standards

| Concern | Rule | Why |
|---------|------|-----|
| Module file length | Soft max ~100 lines | Existing modules are 10–45 lines; growth beyond ~100 is a sign it should be split |
| Module granularity | One logical concern per file | Makes it easy to enable/disable features by commenting out an import |
| `inputs` access | Only declare `inputs` in module args if the module actually uses a flake input | Keeps unused args out of signatures |
| Package management | Prefer `home.packages` for user tools; use `programs.<name>` when home-manager has a module | `programs.*` modules generate correct config file paths automatically |
| Raw configs | Use `xdg.configFile.<name>.source = "${configs}/<dir>"` to symlink entire dirs | Keeps AstroNvim / Niri configs editable without a full rebuild |

**Naming:**
- Module files: lowercase, hyphen-separated (e.g., `zen-browser.nix`, `cpp-dev.nix`)
- Scope labels in commits: lowercase, match the module or area (e.g., `home`, `system`, `gaming`, `font`)

## Decision Tables

### Where to add a new package

| Condition | Where to add | Reason |
|-----------|-------------|--------|
| User-space CLI tool with no config | `home.packages` in `home/default.nix` | Simple one-liner, no dedicated module needed |
| User-space tool with non-trivial config | New `home/modules/<name>.nix`, imported in `home/default.nix` | Keeps default.nix clean |
| Tool provided by a flake input (not nixpkgs) | New or existing `home/modules/<name>.nix` with `inputs` in args | Flake-sourced packages must reference `inputs.<name>.packages.${pkgs.stdenv.hostPlatform.system}.<pkg>` |
| System-level service or hardware config | `laptop/configuration.nix` or new `modules/<name>.nix` | home-manager can't configure system services |
| Font | `modules/fonts/font.nix` | Fonts are system-level (fontconfig) |

### Nix-managed settings vs raw config file in `configs/`

| Condition | Approach | Reason |
|-----------|----------|--------|
| home-manager has a `programs.<name>` module | Use `programs.<name>.settings` / `programs.<name>.enable` | Type-checked, no manual path management |
| Tool has complex config (Lua, KDL) that home-manager can't fully express | Put in `configs/<name>/`, symlink via `xdg.configFile` | Avoids awkward string escaping; config is directly editable |
| Niri window rules / keybindings | `configs/niri/config.kdl` | KDL format not suitable for Nix inline |
| Neovim / AstroNvim lua | `configs/nvim/` | Complex Lua not worth inlining |

### When to create a new flake input vs use nixpkgs

| Condition | Choice | Reason |
|-----------|--------|--------|
| Package exists in nixpkgs unstable | Use nixpkgs | Simpler, benefits from nixpkgs cache |
| Package not in nixpkgs or needs unreleased version | Add flake input with `inputs.nixpkgs.follows = "nixpkgs"` | Ensures consistent nixpkgs version across inputs |
| External project provides a home-manager module (like noctalia, zen-browser) | Flake input | The module API requires the flake |

## Common Pitfalls

### Forgetting `inputs` in module function args
**Root cause:** Modules auto-receive `config`, `pkgs`, `lib`, etc. from home-manager/NixOS, but `inputs` is only available if passed via `extraSpecialArgs`/`specialArgs` — and you must explicitly destructure it.
**Symptom:** `error: attribute 'inputs' missing` at eval time.
**Fix:** Add `inputs` to the module's function args: `{ pkgs, inputs, ... }:`. The flake already passes it via `specialArgs`/`extraSpecialArgs`.

### Editing deployed dotfiles instead of the source
**Root cause:** `~/.config/niri/config.kdl` and `~/.config/nvim/` are symlinks managed by home-manager pointing into the Nix store (which in turn comes from `configs/`). Any direct edit is either ignored or lost on next switch.
**Symptom:** Changes don't persist after `nixos-rebuild switch`.
**Fix:** Edit the source files in `configs/niri/` or `configs/nvim/`, then rebuild.

### Wallpaper path relies on the /etc/nixos symlink
**Root cause:** `noctalia.nix` hardcodes the wallpaper path as `/etc/nixos/home/wallpapers/...`. This works because `/etc/nixos` is a symlink to `/home/dnkyr/nixos-config`.
**Symptom:** Wallpaper not found if the repo is ever moved or the symlink breaks.
**Fix:** If the repo moves, update the path in `home/modules/noctalia.nix` and recreate the `/etc/nixos` symlink.

### `useOSProber = true` requires other OS to be mounted at boot
**Root cause:** GRUB's OS prober scans for other bootable partitions. If Windows or another OS partition isn't visible during build, it may not appear in GRUB.
**Symptom:** Dual-boot entry disappears from GRUB menu.
**Fix:** This is expected behavior; rebuild while the partition is accessible.

## Workflow

**Setup (fresh clone):**
```bash
# Symlink repo to /etc/nixos (requires root)
sudo ln -s /home/dnkyr/nixos-config /etc/nixos

# Apply the configuration
cd /home/dnkyr/nixos-config
sudo nixos-rebuild switch
```

**Making changes:**
1. Edit the relevant `.nix` file (see Architecture above for which layer)
2. Test with `sudo nixos-rebuild test` (activates but doesn't set as boot default)
3. If good, `sudo nixos-rebuild switch` (sets as default)
4. For raw dotfiles (`configs/`), edit directly — no rebuild needed for most editors, but Niri requires a compositor reload

**Commit conventions:**
```
ACTION(scope): short description
```
- `ACTION` ∈ `ADD`, `MODIFY`, `REMOVE`, `FIX`
- `scope` = affected module/area in lowercase (e.g., `home`, `system`, `gaming`, `font`, `shell`, `niri`)
- Examples: `ADD(home): install yazi file explorer`, `MODIFY(font): set system default sans`, `FIX(shell): correct zsh alias for lsd`

**Flake input updates:**
```bash
nix flake update          # update all inputs
nix flake update nixpkgs  # update a single input
sudo nixos-rebuild switch # apply
```
