# nixos-config

> DNKYr's NixOS flake managing two hosts declaratively: `aether` (AMD laptop, full desktop) and `ikuyo` (Hetzner VPS, headless server running a Cloudflare tunnel + Xray VLESS proxy). System config + home-manager + agenix-managed secrets, all assembled via flake-parts and import-tree.

---

## Quick Reference

| Task | Command |
|------|---------|
| Rebuild & switch (current host) | `sudo nixos-rebuild switch --flake .` (from `/home/dnkyr/nixos-config`) |
| Rebuild a specific host | `sudo nixos-rebuild switch --flake .#aether` (or `.#ikuyo`) |
| Rebuild via just (aether) | `just aether` |
| Rebuild via just (ikuyo) | `just ikuyo` (remote, uses `--target-host ikuyo --sudo --ask-sudo-password`) |
| Update flake inputs via just | `just up` |
| List all just recipes | `just` or `just --list` |
| Test without switching | `sudo nixos-rebuild test --flake .` |
| Build without activating | `sudo nixos-rebuild build --flake .` |
| Update flake inputs | `nix flake update` |
| Garbage collect | `sudo nix-collect-garbage -d` (automated weekly via `modules/features/optimize.nix`) |
| Edit an agenix secret | `cd secrets && agenix -e <name>.age` |
| Re-key secrets after key change | `cd secrets && agenix -r` |
| Provision a fresh ikuyo VPS | `nix run github:nix-community/nixos-anywhere -- --flake .#ikuyo --target-host root@<ip>` |

**Non-negotiables:**
- Never edit `modules/hosts/<host>/hardware.nix` — auto-generated by `nixos-generate-config` (aether) or by nixos-anywhere (ikuyo).
- Always declare `inputs` in home-manager module function args when the module needs a flake input (e.g., `nix-gaming`, `zen-browser`, `noctalia`, `emtodo`) — forgetting this causes an eval error.
- Feature modules under `modules/features/` MUST follow the flake-parts pattern (`{ ... }: { flake.nixosModules.<name> = { ... }: { ... }; }`) and be imported by name in the host's `configuration.nix` — `import-tree` only registers them, it does not auto-activate them.
- Never bump `home.stateVersion` or `system.stateVersion` without reading the NixOS release notes — these are one-way migrations that can break stateful data.
- Raw configs (`configs/niri/`, `configs/nvim/`) are symlinked by home-manager; edit the source in `configs/`, not the deployed path in `~/.config/`.
- The flake depends on a **private** input `inputs.secrets` (`git+ssh://git@github.com/DNKYr/nixos-config-secrets`) for non-secret-but-private values (cloudflared tunnel ID + hostname). Eval will fail without SSH access to that repo — that's intentional, not a bug. Use it for any value you want kept out of the public history that isn't sensitive enough to need agenix encryption.
- `secrets/secrets.nix` lists the host SSH public keys allowed to decrypt each `.age` file. After adding a new host or rotating a key, update this file AND re-key (`agenix -r`) — otherwise the host can't read its own secrets at activation time.
- Hibernate on aether is currently **disabled** — `self.nixosModules.aether-hibernate` is commented out in `modules/hosts/aether/configuration.nix`. The module (`hibernate.nix`) is retained for easy re-enabling. If re-enabled, the swapfile-based setup hardcodes `boot.resumeDevice` UUID and `resume_offset`; if `/swapfile` is recreated, recompute the offset (`sudo filefrag -v /swapfile | awk 'FNR==4{print $4}' | sed 's/\.\.//'`) before rebuilding, or hibernate-resume will silently fail.
- `ikuyo` uses GRUB with `efiSupport = false` (BIOS boot via the GPT EF02 partition declared in `disko.nix`). Don't flip on `efiSupport` without also reshaping the disk layout.
- `ikuyo`'s firewall opens **only** TCP 22 (SSH) and TCP 443 (Xray VLESS+Reality direct inbound). The Cloudflare tunnel is outbound-only and needs no inbound rule. Adding a new public service means adding the port to `networking.firewall.allowedTCPPorts` in `modules/hosts/ikuyo/configuration.nix`.

## Project Overview

A Nix flakes-based NixOS configuration for two hosts:
- **`aether`** — AMD laptop with full desktop (Niri Wayland, gaming, IME, home-manager). Hibernate-to-swapfile is wired but currently disabled.
- **`ikuyo`** — Hetzner Cloud VPS (qemu-guest, BIOS/GRUB, ext4 via [disko](https://github.com/nix-community/disko), SSH-only login, secrets via [agenix](https://github.com/ryantm/agenix), no home-manager). Runs a Cloudflare tunnel (`services.cloudflared`) fronting `http://localhost:8001`, plus Xray with a VLESS+Reality direct inbound on `:443`.

The flake is assembled with [`flake-parts`](https://flake.parts) and [`import-tree`](https://github.com/vic/import-tree): every `.nix` file under `./modules/` is auto-discovered and merged into the flake. Each system-level module declares itself as `flake.nixosModules.<name>`; each host's `configuration.nix` then composes the desired modules via `self.nixosModules.<name>`. User environment (packages, dotfiles, shell) is layered on top via home-manager (`home/`), kept out of the auto-import tree and pulled in explicitly from the host config — currently only `aether` wires home-manager. External programs without nixpkgs packages are pulled in via flake inputs (noctalia shell, zen-browser, nix-gaming, emtodo). Remote provisioning uses `nixos-anywhere` + `disko`; secrets are encrypted at rest with `agenix`; non-secret-but-private values (tunnel IDs, hostnames) live in a separate private flake input (`inputs.secrets`).

**Stack:** Nix flakes, flake-parts, import-tree, NixOS unstable, home-manager, disko, nixos-anywhere, agenix, Niri (Wayland) + SDDM, Zsh + Nushell + Starship, Zellij, Alacritty, Neovim (AstroNvim) + Helix, btop, Jujutsu + Git/gh, fcitx5 (mozc + Chinese addons), Pipewire, Bluetooth, Steam + Gamescope + Gamemode, auto-cpufreq, `clash-verge-rev` (VPN client), openconnect (UCLA VPN on aether), OpenSSH + `cloudflared` + `xray` (ikuyo). Task runner: `just`.
**Entry point:** `flake.nix` → `flake-parts.lib.mkFlake` → `import-tree ./modules` → `nixosConfigurations.{aether,ikuyo}`.

**Key invariants:**
- `modules/` is auto-discovered by `import-tree`. Any `.nix` file dropped under it is imported as a flake-parts module — keep it free of dead code.
- Feature modules under `modules/features/` follow the pattern `{ ... }: { flake.nixosModules.<name> = { ... }: { ...NixOS options... }; };`. To consume one, add `self.nixosModules.<name>` to `imports` in the relevant host's `configuration.nix`.
- Each host gets its own subdir under `modules/hosts/<host>/` with `default.nix` (declares `flake.nixosConfigurations.<host>`) + `configuration.nix` (declares `flake.nixosModules.<host>-config`) + any host-specific helper modules (`disko.nix`, `hibernate.nix`, `xray.nix`, `cloudflared.nix`, …). Cross-host code lives in `modules/features/`.
- Home-manager is wired per-host in `modules/hosts/<host>/default.nix`; `home/` files are NOT auto-imported. New home modules go under `home/modules/core/` (terminal essentials), `home/modules/cli/` (currently the AI-tools bucket), or `home/modules/gui/` (needs display server), and are pulled in by name from the host's `home/hosts/<host>/home.nix`. Only `aether` currently uses home-manager — `ikuyo` is system-only.
- `modules/` is for NixOS system-level options only; user-space belongs in `home/modules/`.
- `configs/` is for raw non-Nix config files only; Nix-managed program settings go inline in the `.nix` module.
- `flake.nix` passes `inputs` to the flake-parts tree (outer-module args `{ self, inputs, ... }`) and to home-manager via `extraSpecialArgs = { inherit inputs; }` — home modules destructure `inputs` directly.
- `secrets/secrets.nix` is the access-control list: every `.age` file must list the SSH public keys (from `/etc/ssh/ssh_host_ed25519_key.pub` of each consuming host, plus your editing key) allowed to decrypt it. After editing, run `agenix -r` to re-encrypt with the current key set.
- `inputs.secrets` (the private SSH flake input) is for **non-sensitive but private** values (tunnel IDs, hostnames you don't want in public git history). Anything sensitive (passwords, keys, tunnel credentials JSON) belongs in `secrets/*.age` instead — those stay encrypted at rest, while `inputs.secrets` is plaintext inside a private repo.

## Decision Tables

### Where to add a new package

| Condition | Where to add | Reason |
|-----------|-------------|--------|
| User-space terminal essential (editor, shell, multiplexer, btop-style monitor) | `home/modules/core/` (existing subdir or new file) | The `core/` bucket is the "any-host" terminal baseline, including headless |
| User-space CLI/TUI that doesn't fit `core/` cleanly (AI tools, niche utilities) | `home/modules/cli/` | Kept separate from `core/` for grouping; still terminal-only |
| User-space GUI app (needs display server) | `home/modules/gui/` (new file or existing) | Keeps GUI deps out of headless hosts |
| User-space app/utility with no config | `home.packages` in `home/base.nix` (cross-host) or in the appropriate `core/`/`cli/`/`gui/default.nix` |  |
| User-space app specific to `aether` only | `home.packages` in `home/hosts/aether/home.nix` | Keeps shared modules portable |
| User-space tool with non-trivial config | New `home/modules/{core,cli,gui}/<name>.nix`, imported via the matching `default.nix` | Keeps host file thin |
| Tool provided by a flake input (not nixpkgs) | New or existing `home/modules/{core,cli,gui}/<name>.nix` with `inputs` in args | Flake-sourced packages reference `inputs.<name>.packages.${pkgs.stdenv.hostPlatform.system}.<pkg>` (see `home/modules/gui/gaming.nix`) |
| System-level service or hardware config (reusable across hosts) | New `modules/features/<name>.nix` defining `flake.nixosModules.<name>`, then add `self.nixosModules.<name>` to the consuming host's `configuration.nix` imports | flake-parts auto-discovers the file; each host explicitly opts in |
| System-level config truly specific to one host | Inline in `modules/hosts/<host>/configuration.nix`, or a new `modules/hosts/<host>/<topic>.nix` declaring `flake.nixosModules.<host>-<topic>` | Host-private, not reusable (see `ikuyo-xray`, `ikuyo-cloudflared`) |
| Font | `modules/features/font.nix` | Fonts are system-level (fontconfig) |
| Encrypted credential / password / API key / service config containing secrets | New `secrets/<name>.age` (via `agenix -e`), referenced as `age.secrets.<name>.file = ../../../secrets/<name>.age;` and consumed via `config.age.secrets.<name>.path` | Plaintext secrets must never enter the Nix store / git history |
| Non-sensitive but private value (tunnel ID, hostname, internal URL) | Push to the `nixos-config-secrets` private repo and consume via `inputs.secrets.lib.*` | Keeps the public flake clonable for inspection without leaking infra details |

### Nix-managed settings vs raw config file in `configs/`

| Condition | Approach | Reason |
|-----------|----------|--------|
| home-manager has a `programs.<name>` module | Use `programs.<name>.settings` / `programs.<name>.enable` | Type-checked, no manual path management |
| Tool has complex config (Lua, KDL) that home-manager can't fully express | Put in `configs/<name>/`, symlink via `xdg.configFile` | Avoids awkward string escaping; config is directly editable |
| Niri window rules / keybindings | `configs/niri/config.kdl` | KDL format not suitable for Nix inline |
| Neovim / AstroNvim lua | `configs/nvim/` | Complex Lua not worth inlining |

### When to create a new flake input vs use nixpkgs

| Condition | Choice | Reason |
|-----------|--------|--------|
| Package exists in nixpkgs unstable | Use nixpkgs | Simpler, benefits from nixpkgs cache |
| Package not in nixpkgs or needs unreleased version | Add flake input with `inputs.nixpkgs.follows = "nixpkgs"` | Ensures consistent nixpkgs version across inputs |
| External project provides a home-manager / NixOS module (noctalia, zen-browser, agenix, disko) | Flake input | The module API requires the flake |
| Private values you control (own infra metadata) | Extend the `secrets` private repo, consume via `inputs.secrets.lib.*` | One place for all private-but-non-encrypted values |

### Adding a new host

| Step | Action |
|------|--------|
| 1. Create `modules/hosts/<host>/default.nix` | Declare `flake.nixosConfigurations.<host> = inputs.nixpkgs.lib.nixosSystem { modules = [ self.nixosModules.<host>-config ... ]; };` |
| 2. Create `modules/hosts/<host>/configuration.nix` | Declare `flake.nixosModules.<host>-config = { ... }: { imports = [ self.nixosModules.<host>-hardware ... ]; ... };` |
| 3. Generate hardware config | On the target machine: `nixos-generate-config --root /mnt --no-filesystems`, then move to `modules/hosts/<host>/hardware.nix` and wrap in the flake-parts pattern (`flake.nixosModules.<host>-hardware = ...`) |
| 4. (Optional) Add disko config | `modules/hosts/<host>/disko.nix` declaring `flake.nixosModules.<host>-disko` (used by nixos-anywhere) |
| 5. (Optional) Wire home-manager | In `default.nix`, add `inputs.home-manager.nixosModules.home-manager` + `home-manager.users.<user> = import ../../../home/hosts/<host>/home.nix;` |
| 6. Add the host's SSH pubkey to `secrets/secrets.nix` | Required if the host needs to decrypt any agenix secret |
| 7. (Optional) Add a `just` recipe for the new host | Add a recipe in `justfile` following the `aether`/`ikuyo` pattern |

## Common Pitfalls

### Forgetting `inputs` in a home-manager module
**Root cause:** Home modules auto-receive `config`, `pkgs`, `lib`, etc., but `inputs` is only available because the host wires `home-manager.extraSpecialArgs = { inherit inputs; }` in `modules/hosts/aether/default.nix`. You still have to destructure it.
**Symptom:** `error: attribute 'inputs' missing` at eval time.
**Fix:** Add `inputs` to the module's function args: `{ pkgs, inputs, ... }:`. See `home/modules/gui/gaming.nix`, `noctalia.nix`, `zen-browser.nix`, and `home/hosts/aether/home.nix` for examples.

### Editor LSPs are shared in `editors/default.nix`, not per-editor
**Root cause:** LSPs and formatters (nil, nixpkgs-fmt, lua-language-server, pyright, clang-tools, rust-analyzer) live in `home/modules/core/editors/default.nix` as shared `home.packages` consumed by both helix and neovim. They are NOT in `neovim.nix` itself or managed by AstroNvim.
**Symptom:** Adding an LSP to `neovim.nix` works but duplicates it; removing one from AstroNvim config doesn't remove it from the system because it's installed via the shared Nix list.
**Fix:** When adding or removing an LSP/formatter that both editors should share, edit `home/modules/core/editors/default.nix`. If it's editor-specific, add it to the individual editor file instead.

### Wrong outer args in a feature module (flake-parts vs NixOS)
**Root cause:** Feature modules under `modules/features/` and host modules under `modules/hosts/<host>/` are flake-parts modules, not NixOS modules. The outer function receives flake-parts args (`{ self, inputs, ... }`) and must declare `flake.nixosModules.<name> = { ...NixOS args... }: { ... };`. Trying to put NixOS options at the top level (or destructuring `pkgs` in the outer args) won't work.
**Symptom:** `error: attribute 'pkgs' missing` in the outer function, or options like `services.X` being silently ignored.
**Fix:** Match the pattern in existing files (`modules/features/niri.nix`, `optimize.nix`, `modules/hosts/ikuyo/disko.nix`, `xray.nix`, `cloudflared.nix`): outer `{ ... }:` returns `{ flake.nixosModules.<name> = { config, pkgs, ... }: { ... }; }`. Then add `self.nixosModules.<name>` to the consuming host's imports.

### Adding a feature module but forgetting to import it
**Root cause:** `import-tree` discovering a file under `modules/` only registers it (e.g. as `flake.nixosModules.foo`). No host actually uses it until you list `self.nixosModules.foo` in the `imports` of the relevant host's `configuration.nix` (`aether-config` or `ikuyo-config`).
**Symptom:** New module's options have no effect after rebuild; no eval error.
**Fix:** Add the entry to the consuming host's `imports = [ ... ]` list.

### New `.nix` files are invisible to `nix build` until git tracks them
**Root cause:** Flake eval uses git's view of the working tree by default. A brand-new `.nix` file under `modules/` is untracked, so `import-tree` doesn't see it and the change appears to have no effect.
**Symptom:** New file exists, no eval error, but rebuilds behave as if it weren't there.
**Fix:** `git add -N path/to/new.nix` (or a full `git add`) so the file enters git's index, then rebuild.

### Putting a GUI-dependent home module under `home/modules/core/` or `cli/`
**Root cause:** The `core/`+`cli/` vs `gui/` split exists so headless hosts can import only the terminal set without pulling in a display server. A module that quietly depends on Wayland/X11 (e.g. references `pkgs.alacritty`, sets `wayland.windowManager.*`) breaks that contract.
**Symptom:** Closure size balloons on a headless host, or `nix build` pulls in graphics stacks unexpectedly.
**Fix:** Move the offender to `home/modules/gui/` and import it from `gui/default.nix`. Reserve `core/` and `cli/` for tools that work in a TTY.

### Editing deployed dotfiles instead of the source
**Root cause:** `~/.config/niri/config.kdl` and `~/.config/nvim/` are symlinks managed by home-manager pointing into the Nix store (sourced from `configs/`, wired via `xdg.configFile."<name>".source` in `home/hosts/aether/home.nix`). Direct edits are either ignored or lost on next switch.
**Symptom:** Changes don't persist after `nixos-rebuild switch`.
**Fix:** Edit the source files in `configs/niri/` or `configs/nvim/`, then rebuild.

### Wallpaper path relies on the `/etc/nixos` symlink
**Root cause:** `home/modules/gui/noctalia.nix` hardcodes the wallpaper path as `/etc/nixos/home/wallpapers/1387138.png`. This works because `/etc/nixos` is a symlink to `/home/dnkyr/nixos-config`. Wallpaper images live in `home/wallpapers/` (currently: `1387138.png`, `Anime-Girl-Rain.png`, `cirno-nixos.jpg`).
**Symptom:** Wallpaper not found if the repo is ever moved, the symlink breaks, or the referenced filename is renamed.
**Fix:** If the repo moves, update the path in `home/modules/gui/noctalia.nix` and recreate the `/etc/nixos` symlink. To swap wallpapers, change the filename in that module (don't expect the deployed config to update if you only rename the file).

### `useOSProber = true` requires other OS to be mounted at boot
**Root cause:** GRUB's OS prober scans for other bootable partitions. If Windows or another OS partition isn't visible during build, it may not appear in GRUB.
**Symptom:** Dual-boot entry disappears from GRUB menu.
**Fix:** This is expected behavior; rebuild while the partition is accessible.

### agenix secret unreadable on a host after key rotation or new host
**Root cause:** Each `.age` file is encrypted to the exact set of public keys in `secrets/secrets.nix` at the time of `agenix -e`. If a host's SSH host key changes (reinstall, fresh deploy), or a new host is added, existing secrets stay encrypted to the old key set.
**Symptom:** Activation fails on the host with `agenix: failed to decrypt /run/agenix/<name>` or the user has no usable password / xray won't start / cloudflared can't authenticate.
**Fix:** Update `secrets/secrets.nix` with the current host pubkey (`cat /etc/ssh/ssh_host_ed25519_key.pub` on the host), then `cd secrets && agenix -r` to re-encrypt every secret to the new key set. Commit the updated `.age` files.

### Cloning the public repo fails eval because `inputs.secrets` is private
**Root cause:** `flake.nix` declares `secrets` as a `git+ssh://` flake input pointing at a private GitHub repo. Without SSH access to `DNKYr/nixos-config-secrets`, `nix flake` operations fail at the input-fetch stage.
**Symptom:** `error: unable to download ...nixos-config-secrets...: Permission denied (publickey)`.
**Fix (for the owner):** Ensure SSH key has access to the repo. **Fix (for anyone else):** Either get read access, or fork the flake and replace `inputs.secrets` with a stub that provides the same `lib.ikuyo.cloudflared.{tunnelId,hostname}` shape.

### nixos-anywhere wipes the disk per `disko.nix`
**Root cause:** `nixos-anywhere --flake .#ikuyo` runs the `ikuyo-disko` module, which formats `/dev/sda` end-to-end (GPT, EF02 boot, ext4 root). This is the intended bootstrap flow for a fresh VPS but is destructive on any existing data.
**Symptom:** All data on the target host is gone after a single command.
**Fix:** Only run `nixos-anywhere` against a host you actually intend to wipe. For routine updates of an already-installed `ikuyo`, use `just ikuyo` (or `sudo nixos-rebuild switch --flake .#ikuyo --target-host root@<ip>`) instead — that respects the existing filesystem.
